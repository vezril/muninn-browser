# Task 2.1 evidence — Pass extension bundle acquisition (feeds AS-8 ADR)

*Desk research executed 2026-07-11/12 by a research agent (live endpoint checks + repo inspection). Recorded verbatim below; recommendation feeds ADR-001.*

## Headline discovery that reframes the question

The five options are **not interchangeable artifacts** — they yield different *build targets*, and the build target determines whether the login fork (Spike B risk #1) works out of the box:

- **`packages/shared/lib/browser/extension.ts`** (the account.proton.me side of the handshake): `sendMessageSupported()` requires `browserAPI?.runtime?.sendMessage` to exist *on the web page*. In Muninn's WKWebView, `window.chrome.runtime` does not exist, so account.proton.me **automatically falls back to `window.postMessage`** (token-correlated, `from: PROTONACCOUNT`) — the same path it uses for Firefox. The extension-ID broadcast list (`PROTONPASS_EXTENSION_IDS` in `packages/shared/lib/constants.ts`: `ghmbeldphafepmbegfdlkpapadhbakde` chromium prod, `hlaiofkbmjenhgeinjlmkafaipackfjh` beta, `gcllgfdnfnllodcaambdaknbipemelie` edge, `me.proton.pass.catalyst.safari-extension (2SB5Z68H26)` Safari) is only used on the `runtime.sendMessage` path — **the fallback path matches on message shape, not extension ID**.
- **The Safari-target bundle ships `fork.js`** (`manifest-safari.json` content script on `https://account.proton.me/*`, source `src/app/content/safari/index.ts`, webpack comment: "Safari fork fallback"), which listens for exactly that postMessage and relays to the background worker. The Firefox bundle ships the equivalent `external.js`. **The Chrome bundle ships neither** — it relies on real `externally_connectable` support, which Muninn would have to emulate with a hand-written bridge.

**Consequence:** acquiring the *Safari-target* bundle largely dissolves Spike B risk #1 — the bridge Muninn planned to write already exists inside Proton's own artifact. The Chrome CRX is the *worst* artifact for Muninn despite being the easiest to download.

## Option 1 — Chrome Web Store CRX download

**Feasibility: confirmed working, tested live.** `curl -sIL "https://clients2.google.com/service/update2/crx?response=redirect&prodversion=126.0&acceptformat=crx3&x=id%3Dghmbeldphafepmbegfdlkpapadhbakde%26uc"` → HTTP 302 → `clients2.googleusercontent.com/.../GHMBELDPHAFEPMBEGFDLKPAPADHBAKDE_1_38_2_0.crx`, 200 OK, 8,034,956 bytes, `application/x-chrome-extension`, last-modified 2026-07-10. The `response=updatecheck` variant returns Omaha XML with `version="1.38.2"` **and `hash_sha256=2cc54d72218fe09081e8e38261ae0f16aca51c62722828f46a33109f88c7c679`** — i.e., Google's own endpoint provides version discovery + a pinnable hash without downloading.

- **Identity:** CWS-packed CRXs carry the canonical public key; the ID `ghmbeld…` derives from it. Fully preserved. But per above, identity matters less than build target — and this is the **chrome** target: no `fork.js`, MV3 service worker, `externally_connectable`-dependent login, plus Chrome-only baggage (`offscreen.html`, `webRequest` codepaths behind guards).
- **Update mechanics:** poll `updatecheck` XML (version + sha256), fetch on change. CRX3 = protobuf header + ZIP payload; needs unpacking. *Needs spike to verify a clean unpack.*
- **Licensing/ToS:** the extension itself is **GPLv3** (`ProtonMail/WebClients/LICENSE`, verified). The friction is Google's side: no official public CRX download API; downloading at first run on the owner's machine for personal use is common practice and low-risk; **shipping the Google-served CRX inside a public app would be redistribution of a Google-store artifact and should be avoided**.
- **Solo-dev burden:** lowest of all options — but buys the wrong build target.

## Option 2 — Firefox AMO XPI download

**Feasibility: confirmed.** AMO API `https://addons.mozilla.org/api/v5/addons/addon/proton-pass/` returns current version **1.37.2**, public download URL `https://addons.mozilla.org/firefox/downloads/file/4824869/proton_pass-1.37.2.xpi`, GUID `78272b6fa58f4a1abaac99321d503a20@proton.me`, license "GPL-3.0-only". XPIs are plain ZIP; AMO explicitly serves public downloads — cleanest ToS posture of the store options.

- **Build-target fit: mixed.** Includes `external.js` (login fork would work). But `manifest-firefox.json` carries `webRequest`+`webRequestBlocking`, background is an **event page** not a service worker, `USE_IMPORT_SCRIPTS=false` changes chunk-loading. The Firefox codepaths are exactly the ones the shim was *not* specced against.
- **Version lag observed:** 1.37.2 (2026-05-28) vs Chrome's 1.38.2.
- **Verdict:** viable fallback, but the shim spec says "Safari build profile" and this isn't it.

## Option 3 — Building from source (Safari target)

**Feasibility: confirmed, better than the spike assumed.** `applications/pass-extension/README.md`: Yarn monorepo; `BUILD_TARGET=chrome|firefox|safari yarn build:extension` → `dist/`. **The Safari web bundle is a plain webpack build** — the Ruby/Xcode steps only wrap `dist/` into an .appex for Safari proper; **Muninn needs only `dist/`, so Xcode/Ruby are skippable.** Per-release git tags exist (`proton-pass@1.38.2`, `proton-pass@1.38.1`, …) — you can build exactly the shipped version.

- **Identity: preserved when wanted.** The repo checks in **`manifest-keys.json`** with the chrome/edge production public keys, and `webpack.config.ts` (~L343) does `if (PUBLIC_KEY) manifest.key = PUBLIC_KEY` for chrome builds — even a source-built chrome bundle reproduces the canonical `ghmbeld…` ID. For the Safari target this is moot: the fallback handshake is ID-agnostic.
- **Supply chain:** trades Google's signed blob for the npm tree of a large monorepo at a pinned git tag. For a privacy-first project, building from a readable pinned tag is arguably the *stronger* trust story.
- **Licensing: cleanest of all.** GPLv3 source, built yourself — no store ToS at all. **The only option that stays clean if Muninn is ever publicly distributed** (GPLv3: ship license + source offer).
- **Burden:** highest — Node/Yarn toolchain, multi-GB `node_modules`, occasional toolchain breakage. As a *refresh* workflow it's a chore; as an *escape hatch* (patching, debugging, diffing) it's invaluable.

## Option 4 — Extracting from Proton Pass for Safari.app

**Feasibility: confirmed on this very Mac.** The app ([MAS id6502835663](https://apps.apple.com/us/app/proton-pass-for-safari/id6502835663?mt=12)) is already installed at `/Applications/Proton Pass for Safari.app`, and the **complete production Safari-target web bundle sits in plaintext** at `Contents/PlugIns/Safari Extension.appex/Contents/Resources/`: `manifest.json` (version **1.38.0**), `background.js` (`service_worker`), content scripts `[orchestrator.js, fork.js, webauthn.js]`, `popup.html`, `dropdown.html`, `notification.html`, `settings.html`, `onboarding.html`, crypto chunks, assets. MAS FairPlay encryption covers the executable, not `Resources/` — a plain `cp -R` of the web files suffices (skip the `ProtonCore_*.bundle` native payloads).

- **This is the exact artifact the shim was specced against** — Proton's shipped Safari build, `fork.js` included, produced and QA'd by Proton.
- **Version:** 1.38.0 vs 1.38.2 on CWS — the Safari channel lags days-to-weeks (App Store review latency). Acceptable; the Safari build is already the canary per CLAUDE.md.
- **Update mechanics:** MAS auto-updates the app; a refresh script diffs the appex `manifest.json` version against the vendored copy and re-copies. Fully offline.
- **Licensing:** the web bundle is Proton's GPLv3 code; extracting on your own machine for personal use is unproblematic. For **public distribution** you'd switch to Option 3 (build the identical target from source under GPL) rather than redistribute the MAS artifact.
- **Burden:** near-zero, with one dependency: the MAS app installed (free; already present).

## Option 5 — Vendored copy checked into the repo

Not an acquisition source but the **storage discipline** for whichever source is chosen:

- `vendor/pass-extension/<version>/` — the unpacked web bundle, committed (GPLv3 permits). Plus `vendor/pass-extension/MANIFEST.lock` recording: source (safari-appex | cws-crx | source-build @ git tag), version, sha256, extraction date.
- Refresh = `tools/refresh-pass-bundle.sh`: check source for new version → copy/unpack → write lock → `git diff` review of `manifest.json` + entry points → **re-run the Spike B grep inventory (FR-25)** → commit. Version bumps become reviewable diffs; builds are hermetic; Muninn regressions bisectable against extension versions.

## Recommendation

**Hybrid: extract the Safari-target bundle from Proton Pass for Safari.app (Option 4) as the primary acquisition path, vendor it in-repo with a hash lockfile (Option 5), and keep a pinned-tag source build (Option 3, `BUILD_TARGET=safari yarn build:extension`) as the verified escape hatch and the mandatory path if Muninn ever ships publicly.**

1. **Build target beats acquisition convenience.** The Safari bundle is the only artifact whose login-fork mechanics (`fork.js` + account app's postMessage fallback) work in a non-Chrome host without a hand-written `onMessageExternal` bridge — it converts Spike B's #1 risk from "build a bridge" to "inject Proton's own bridge per the manifest."
2. **The MAS app is already on this Mac, in plaintext, at v1.38.0** — extraction is a `cp`, updates ride the App Store, artifact is Proton-produced.
3. **Source build at tag `proton-pass@<version>`** covers diffing/debugging, local patches, closing MAS lag, and the public-distribution scenario. `manifest-keys.json` + webpack key injection makes canonical-ID chrome builds reproducible if the Chromium fallback line is ever exercised.
4. **CWS `updatecheck` endpoint as version telemetry only** — a cheap "Proton shipped something" signal, no CRX downloads.

- **S2 — fork.js handshake end-to-end:** confirm account.proton.me takes the postMessage fallback inside Muninn's WKWebView (verify nothing in the shim leaks `browserAPI.runtime.sendMessage` into page scope; check whether `fork.js` gates on user-agent).
- **S3 — CRX3 unpack** (only if the CWS path is ever needed).
- **S4 — Source-build parity:** build at tag `proton-pass@1.38.0`, diff `dist/` against the appex extraction — quantifies escape-hatch fidelity.
