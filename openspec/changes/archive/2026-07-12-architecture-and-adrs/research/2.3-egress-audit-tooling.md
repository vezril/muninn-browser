# Task 2.3 evidence — Egress-audit tooling shape (feeds NFR-5 ADR)

*Desk research executed 2026-07-11 by a research agent (Apple docs/forums, tool docs, precedent projects — all cited). Recorded verbatim below; feeds ADR-002.*

**Question:** throwaway manual setup vs. small reusable harness for the shell/shim egress audit.
**Short answer:** reusable harness — the audit recurs at every milestone, the hard part (attributing and classifying WKWebView traffic) is exactly the part you don't want to re-derive by hand each time, and the only precedent that stayed useful over time (Brave's `network-audit`) is a checked-in script + allowlist. Recommended shape: **pktap capture (`tcpdump -k`) for attribution + in-app/per-datastore instrumentation for classification, wrapped in a repo script — no TLS interception anywhere.**

## 0. Framing: what NFR-5 actually requires

The audit is a **host-level allowlist check** over shell/shim-originated flows. Evidence needed per connection: `(destination host, originating class ∈ {shell-native, shim/background-host, page})`. It does **not** require payload visibility — TLS interception is never necessary, which satisfies the credential ground rule *by construction*.

Two hard sub-problems drive the tooling choice:

- **Attribution** (which process): WKWebView networking egresses from the shared `com.apple.WebKit.Networking` XPC service (shared across every WebKit app). Apple DTS ("Quinn") confirms there is virtually no API surface to resolve responsibility — except Endpoint Security's `responsible_audit_token` and **pktap** ([Apple forums 739414](https://developer.apple.com/forums/thread/739414)).
- **Classification** (shell/shim vs page-initiated): no network-level tool can distinguish the shim's background host fetching api.proton.me from a page subresource doing the same — classification must come from **inside** the app, via which `WKWebsiteDataStore` each WebView uses.

## 1. mitmproxy

- **(a) System proxy — rejected.** Whole-Mac capture, no process identity.
- **(b) mitmproxy local capture mode (`--mode local:Muninn`, macOS App Proxy Provider)** — attribution-broken for WKWebView: flow metadata identifies `com.apple.WebKit.Networking`, not the responsible app; would miss all WKWebView traffic or capture every WebKit app ([mitmproxy local capture](https://www.mitmproxy.org/posts/local-capture/macos/)).
- **(c) `WKWebsiteDataStore.proxyConfigurations` (macOS 14+) — the right hook, flips the problem.** Per-data-store proxy configuration: pointing the background-host store and the page-tab store at **different local proxy ports turns classification into a port number** — deterministic, structural, app-side debug flag ([Apple docs](https://developer.apple.com/documentation/webkit/wkwebsitedatastore/proxyconfigurations-cdc1)).
- **TLS interception:** Apple services fail closed under HTTPS interception (OS-level cert pinning, [Apple 101555](https://support.apple.com/en-us/101555)); decrypting `*.proton.me` is exactly what ground rule 1 forbids. **Recording-only works:** `mitmdump --ignore-hosts '.*' --show-ignored-hosts` logs hosts from ClientHello SNI without decryption. For a host-level audit, recording-only is fully sufficient. Even then, a ~50-line bespoke CONNECT/SOCKS host logger could replace mitmproxy entirely — spike decides.

## 2. Little Snitch / LuLu

- **Little Snitch 6** has a real CLI (`littlesnitch log-traffic`, etc., root + opt-in) and has invested in XPC parent-attribution — but whether CLI output attributes *system-framework* XPC traffic needs a hands-on check. Excellent as Calvin's personal continuous monitor; weaker as the milestone gate (paid, GUI-centric, attribution-in-CLI unverified).
- **LuLu**: no first-party CLI; allow/deny prompt firewall, not an evidence recorder. Not suitable.

## 3. Apple-native

- **`tcpdump` + pktap — the attribution winner.** The `pktap` pseudo-interface tags every packet with process metadata; `-k` displays it. Critically it carries both `proc` (socket owner) **and `eproc`/`epid` (the effective/delegated process)** — real captures show `proc com.apple.WebKit…, eproc Safari:804`, i.e. WebKit XPC traffic attributed back to the responsible app via `PTH_FLAG_PROC_DELEGATED` ([tcpdump man](https://manp.gs/mac/1/tcpdump), [nuxx.net pktap guide](https://nuxx.net/blog/2021/10/29/network-capture-with-process-name-and-pid-on-macos/)). **Wireshark/tshark ≥ 4.6.0 parses pktap metadata natively** ([nuxx.net](https://nuxx.net/blog/2025/10/14/wireshark-4-6-0-supports-macos-pktap-metadata-pid-process-name-etc/)). So: `sudo tcpdump -i pktap,en0 -k NP -w session.pcapng`, post-process to `(host ← DNS/SNI, proc, eproc, pid)`. Caveats: sudo; hostnames from DNS/SNI (ECH/DoH would blind it — neither applies to Proton/Apple hosts today; note as assumption).
- **`nettop`**: spot-check only (IP:port, shared-XPC display problem).
- **Network Extension (NEFilterDataProvider)**: confirmed overkill — still can't resolve responsibility without Endpoint Security.
- **Unified logging**: no stable per-connection hostname records; skip.

## 4. In-app instrumentation

- `WKNavigationDelegate` sees navigations only — subresources invisible by design; WebKit has no NetLog equivalent ([Apple forums 110312](https://developer.apple.com/forums/thread/110312)).
- Background-host fetches are ordinary WKWebView networking — invisible to delegates but **fully routable via that store's `proxyConfigurations`**.
- Shell-native `URLSession` calls are Muninn-authored: a debug-build `AuditLogger` wrapper or `connectionProxyDictionary` covers them completely.
- **Classification reliability ranking:** per-datastore proxy routing (deterministic) > native URLSession hooks (deterministic) > delegate heuristics (incomplete — don't build on these). In-app instrumentation is the only reliable classifier but can't be the only *recorder* — the audit must also prove the instrumentation didn't miss anything. Hence the hybrid.

## 5. Precedent

- **Leith, "Web Browser Privacy" (IEEE Access 2021)**: mitmdump transparent proxy + root cert — needed payload MITM because his question was *what data*; ours is only *where* — we skip his hardest, most credential-adjacent step ([paper](https://www.scss.tcd.ie/Doug.Leith/pubs/browser_privacy.pdf)).
- **Brave's `npm run network-audit`**: checked-in CI script — `--log-net-log` + allowlist + checker, failing on unknown hosts; needed allowlist exceptions for platform traffic (the same "Apple system services" category NFR-5 anticipates) ([#19527](https://github.com/brave/brave-browser/issues/19527)). **The closest analog to NFR-5, and emphatically a reusable harness.** WebKit has no NetLog, so Muninn substitutes pktap + proxy-port routing for the event source; the artifact shape (script + allowlist + report, red/green) copies Brave.

## Recommendation

**Build a small reusable harness (`audit/` in the repo), hybrid of §3 + §4. No TLS MITM anywhere.**

Architecture — two independent layers that must agree:
1. **Ground truth / attribution (outside the app):** `tcpdump -i pktap,<iface> -k NP -w session.pcapng`; post-process with tshark ≥ 4.6 to `(host, proc, eproc, pid)`; keep Muninn-attributed rows (PID-set snapshot as fallback).
2. **Classification (inside the app, `MUNINN_EGRESS_AUDIT=1`):** background-host store → localhost:**A**; page-tab store → localhost:**B**; native URLSession → localhost:**C**. Ports A/C gated against the allowlist; B recorded-only.
3. **Checker:** PASS iff every host on A/C ∈ allowlist AND every pcap-attributed flow is explained by exactly one port/log — an unexplained flow is automatic FAIL (instrumentation coverage broke).

Checked-in artifacts: `audit/egress-allowlist.txt` (annotated, per-host rationale), `audit/run-egress-audit.sh` (includes the GUI-launch warning gate per ground rule 2), `audit/session-script.md` (fixed scenario; login performed by Calvin per ground rule 1), generated `audit/report-YYYY-MM-DD.md`. Raw `.pcapng` is **not** committed (browsing record) — deleted post-report.

Sensitive-payload hard rules (encoded in the script): no TLS interception ever (no CA generated; `ignore-hosts '.*'` hardcoded); pcap local-only, gitignored, deleted; AuditLogger logs method + host only for Proton hosts (no paths/queries/bodies/headers); login/unlock human-only, never automated or screenshotted.

**Effort:** spike ½–1 day (verify `eproc` attribution for WebKit.Networking on macOS 26 — the one load-bearing unverified fact; verify `proxyConfigurations` routes service-worker fetches; pick mitmdump vs bespoke logger; toy app opens a window → GUI warning applies). Harness v1: 1–2 days. Hooks land with E1. Per-milestone rerun: ~30 min.

**Fallback if `eproc` attribution fails the spike:** snapshot Muninn's process tree PIDs at launch, filter pcap on PID set; classification layer unaffected.
