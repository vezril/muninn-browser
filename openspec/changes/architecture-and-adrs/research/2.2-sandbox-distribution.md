# Task 2.2 evidence — App Store sandbox vs. Developer ID vs. unsigned personal builds (feeds OQ-5 distribution ADR)

*Desk research executed 2026-07-11 by a research agent (Apple docs, App Review Guidelines, developer writeups — all cited). Recorded verbatim below; feeds ADR-003.*

## 1. App Sandbox entitlements for a browser

The Mac App Store **mandates** App Sandbox (Guideline 2.4.5(i): *"They must be appropriately sandboxed…"*).

Entitlement set a sandboxed Muninn would need — all exist and are MAS-compatible:

| Entitlement | Purpose | Status |
|---|---|---|
| `com.apple.security.app-sandbox` | Sandbox opt-in (MAS-mandatory) | Standard |
| `com.apple.security.network.client` | Outgoing connections. **Required by WKWebView itself** — even for bundle-local content (out-of-process helper loads) | Confirmed (Apple forums 116359; S. Gamel writeup) |
| `com.apple.security.files.downloads.read-write` | Write to `~/Downloads` without a save panel | Standard |
| `com.apple.security.files.user-selected.read-write` | Open/Save panels for everything else | Standard |

**Do the shim techniques work under App Sandbox?** Yes, with one architecture-dependent caveat:

- **WKWebView under sandbox: confirmed working** — it is the sandbox-designed web view.
- **WKURLSchemeHandler, WKContentWorld, WKUserScript, WKScriptMessageHandler:** in-process API surface on `WKWebViewConfiguration` with no entitlement gates documented anywhere — no reported sandbox breakage found (confirmed-by-absence; marked *inferred* in the ledger).
- **Hidden always-alive WKWebView (background host):** no sandbox-specific restriction found; the real risks are process-lifetime ones (App Nap, WebContent termination) that apply identically under both distribution models.
- **What genuinely hurts under sandbox:** container-relocated profile storage, no arbitrary filesystem writes outside entitled scopes, restricted automation, and no private API (historically decisive for Chromium browsers — moot for WebKit-based Muninn).

## 2. Default browser eligibility on macOS

- **Mechanism:** declare `http`/`https` in `CFBundleURLTypes`; macOS lists the app in System Settings → Default web browser. Programmatic: `NSWorkspace.setDefaultApplication(at:toOpenURLsWithScheme:)`.
- **The managed `com.apple.developer.web-browser` entitlement is an iOS 14+ artifact** — no evidence it exists or is required on macOS, and decisive practical evidence it isn't: Arc, Orion, Brave, Chrome, Firefox are all Developer ID direct-download macOS apps and all settable as default.
- **Conclusion: default-browser status on macOS is available to MAS, Developer ID, and locally-built unsigned apps alike.** Info.plist declaration, not a distribution-channel privilege. Confirms FR-29's approach.

## 3. App Review risk (MAS only)

- **2.5.6** (WebKit requirement): Muninn is WKWebView — satisfied trivially. Moot.
- **4.2 Minimum Functionality:** a full custom-shell browser passes comfortably. Low risk.
- **5.2.1 / 5.2.2 — the real MAS risk:** *"Apps should be submitted by the person or legal entity that owns or has licensed the intellectual property…"* / *"If your app uses… content from a third-party service, ensure that you are specifically permitted to do so under the service's terms of use. Authorization must be provided upon request."* Muninn **embeds the Proton Pass extension bundle it doesn't own** and integrates with Proton's service. GPLv3 covers code redistribution, but 5.2.1 trademark exposure ("Proton Pass" branding) and 5.2.2's authorization-on-demand are exactly the hooks a reviewer would use. **The single strongest MAS-specific rejection vector — and entirely absent from the Developer ID path** (notarization is automated malware scanning, not guideline review).

## 4. Developer ID / notarization path

- **Requirements:** paid Apple Developer Program (**US$99/yr**), Developer ID Application cert (paid Account Holders only), hardened runtime + notarization (mandatory since macOS 10.14.5).
- **⚠️ Calvin's enrollment is LAPSED** (recorded 2026-07-11, task 2.4). Renewal is a prerequisite for any Developer ID signing — but **not** needed for v0.x personal use (below).
- **Hardened runtime & JIT:** `com.apple.security.cs.allow-jit` is needed when **JavaScriptCore runs in-process** (Apple DTS/Quinn, forums 674497). **WKWebView does not need it in the host app** — JIT of untrusted JS happens in the separate WebContent XPC process (WebKit Introduction.md). *ADR-relevant nuance:* the shim's background host is spec'd as "hidden WKWebView **or JSContext**" — an in-process `JSContext` re-opens the allow-jit question under hardened runtime. Prefer the hidden-WKWebView host, or budget the entitlement.
- **Unsigned personal builds — verified:** Gatekeeper only evaluates apps carrying the `com.apple.quarantine` xattr (downloaded software); locally built apps are never quarantined (Apple forums 740680; Lapcat). Apple Silicon requires at-least-ad-hoc signing, which Xcode does automatically ("Sign to Run Locally") or a free Personal Team. Sequoia's Gatekeeper changes affect only quarantined apps. None of Muninn's needed capabilities (network, WKWebView, LocalAuthentication, custom schemes, default-browser listing) require a managed entitlement. **Conclusion: v0.x personal use requires zero Apple Developer Program membership; the lapsed enrollment only becomes blocking at first distribution beyond Calvin's machine.**

## 5. Practical precedent

| Browser | Engine | macOS distribution | MAS? |
|---|---|---|---|
| **Arc** | Chromium | Direct download (arc.net) | No |
| **Orion** (Kagi) | **WebKit + WebExtension support (~70% API coverage)** — Muninn's closest cousin | Direct download (orionbrowser.com); App Store only on iOS | No |
| **Brave** | Chromium | Direct download; open issue about even adopting the macOS sandbox | No (iOS only) |

**No precedent exists of a macOS MAS browser with third-party-extension support.** Orion — the nearest analog — deliberately ships direct-download on macOS. Chromium browsers *cannot* ship on MAS (private API use); doesn't bind WebKit-based Muninn but explains the empty field.

## Summary table: constraint × distribution model

| Constraint | Mac App Store | Developer ID (direct) | Unsigned/ad-hoc (personal, v0.x) |
|---|---|---|---|
| App Sandbox | **Mandatory** | Optional | Optional |
| Hardened runtime | Not required | **Required** (notarization) | Not required |
| WKWebView/scheme handler/content worlds/hidden host | Work (no documented breakage) | Work | Work |
| JIT entitlement | N/A | Not needed for WKWebView; only if host becomes in-process JSContext | Not needed |
| Downloads | Via entitlement | Unrestricted | Unrestricted |
| Default-browser eligibility | Yes | **Yes** (proven: Arc/Orion/Brave) | Yes |
| Guideline review | **Full; 5.2.1/5.2.2 Proton IP+ToS exposure = top rejection risk** | None | None |
| Apple Developer membership | Required ($99/yr) | **Required — Calvin's is LAPSED** | **Not required** |
| Precedent browsers | None with extension support | Arc, Orion, Brave, Chrome, Firefox | — |

## Confirmed vs inferred

**Confirmed:** sandbox mandate + guideline texts (Apple guidelines page); `network.client` required by WKWebView under sandbox; WKWebView JIT out-of-process (WebKit Introduction.md) + Quinn's allow-jit guidance; Developer ID = paid membership + notarization since 10.14.5; Gatekeeper quarantine-only behavior + Apple Silicon ad-hoc signing; CFBundleURLTypes → default-browser list; Arc/Orion/Brave channels; Electron/Chromium MAS private-API rejections.

**Inferred (high confidence):** WKURLSchemeHandler/WKContentWorld/hidden-WKWebView having no sandbox-specific breakage (absence of evidence + API design — a ~1 hr sandboxed-target spike would convert to measured fact); `com.apple.developer.web-browser` being iOS-only; Proton redistribution being license-permitted but trademark/ToS-exposed.

## Verdict on OQ-5

**Nothing challenges the PRD's direct-download resolution; the evidence strengthens it.** The shim would technically survive MAS sandboxing — the killer is App Review 5.2.1/5.2.2 (embedding a third party's branded extension, authorization demandable), plus zero precedent of extension-supporting browsers on the macOS App Store. **ADR refinement:** the "revisit MAS at 1.0" clause should carry an explicit precondition — *written authorization from Proton (5.2.2) or a partnership* — otherwise the revisit is a dead letter. And verified: **v0.x on Calvin's own Mac needs no Apple Developer membership at all.**
